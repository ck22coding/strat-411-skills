# 411 Deck Builder

This skill takes a structured deck spec (produced by the `411-case` skill) and produces a `.pptx` file. It handles visual production only — all argument logic, Pyramid Principle structuring, and strategic reasoning live in `411-case`.

**This skill is fully automated.** There is no human-in-the-loop during slide production. Quality depends entirely on the spec and on the QA checks at the end.

**Boundary rule:** This skill never makes argument decisions. If the spec is ambiguous about what a slide should argue, stop and ask the user — don't infer strategic intent.

---

## Phase 1: Parse Handoff

Announce: "Starting Phase 1: Parsing the deck spec."

### Step 1: Read the Spec

Read the handoff markdown file provided by the user. Extract:

1. **YAML frontmatter** — `case_name`, `recommendation_format`, `scqa` (situation, complication, question, answer), `key_line`, `structure` (inductive/deductive), `plural_noun`, `argument_flow` (list of leadline summaries), `sensitivity_slides` (list of slide numbers)
2. **Argument Summary** — the numbered leadline story
3. **Per-slide sections** — each slide's fields: position, leadline, chart type, chart title, data table, axis labels, data labels, callout title, callout items, footnotes, source, formatting notes
4. **Recommendation slide** — leadline, format, recommendation, supporting reasons, key risks, next steps

### Step 2: Validate

Check every slide for required fields:
- Leadline (must be an analytical statement, not a label)
- Chart type
- Data (table or description)
- Source line
- At least one callout item

Flag any gaps to the user with specific slide numbers and missing fields. Do not proceed until gaps are resolved or the user explicitly accepts them.

### Step 3: Present Summary

Tell the user:
- "Found N slides. Argument flow:"
- List all leadlines in order (numbered)
- "Ready to build?"

Wait for confirmation.

## Phase 2: Design Deck

Announce: "Starting Phase 2: Designing slide layouts."

### Step 1: Read Formatting Standards

Read `~/.claude/skills/411-case/references/slide-best-practices.md` for:
- Slide template zones (leadline, graph title, chart, callout, footnotes, source)
- Chart formatting standards
- Implication writing standards
- Recommendation slide format

### Step 2: Map Slides to Layouts

For each slide, determine the layout pattern based on chart type:
- **Bar/column charts** — chart center-left, callout right
- **Line/area charts** — chart full width, callout below or right
- **Scatter plots** — chart center, callout right
- **Waterfall charts** — chart full width, callout right
- **Tables** — table center, callout below
- **Comparison layouts** — side-by-side panels
- **Text-heavy slides** — structured bullet layout with key message hierarchy

### Step 3: Plan Structural Slides

From the frontmatter, plan:
1. **Title slide** — case name
2. **SCQA intro slide(s)** — situation, complication, question, answer (1–2 slides depending on complexity)
3. **Section-break slides** — one per argument flow item (if inductive structure with 3+ sections)
4. **Recommendation slide** — per the spec's format field

### Step 4: Present Design Choices

Show the user the full slide sequence with layout assignments. Ask: "Does this layout plan look right? Any slides that need a different treatment?"

Wait for approval.

## Phase 3: Produce Slides

Announce: "Starting Phase 3: Building the .pptx."

### Step 1: Enforce Template Zones

Every data slide must follow this structure:
- **Leadline** (top) — analytical takeaway, not a label
- **Graph title** (on the chart) — descriptive label of what the chart shows
- **Chart** (center) — built from the data table in the spec
- **Callout section** (right side) — "Key Implications" or "Key Takeaways"
- **Footnotes** (bottom) — methodology notes, caveats
- **Source line** (bottom) — "Source: [specific source], [year]"

### Step 2: Build Charts

For each slide's data table:
1. Determine the chart type from the spec
2. Build the chart with proper axis labels, data labels per spec instructions
3. Apply consistent formatting (colors, fonts, gridlines)

### Step 3: Apply Sensitivity Format

For slides listed in `sensitivity_slides`:
- Show range of outcomes under different assumption values
- Use scenario bands, tornado charts, or sensitivity tables as appropriate
- Label the base case clearly

### Step 4: Build Recommendation Slide

Per the spec's `recommendation_format`:
- **"single"** — direct recommendation + supporting reasons + key risks + next steps
- **"multiple-options"** — numbered options with "choose this if..." criteria + key risks

### Step 5: Produce .pptx

Use the basic slide template — no fancy formatting, backgrounds, or color themes. Clean and simple.

Invoke the `pptx` skill to produce the final `.pptx` file, using `Basic Slide Template.pptx` (in this skill's directory) as the base template.

---

## Phase 4: QA

Announce: "Starting Phase 4: Quality checks."

### Step 1: Content QA

Use `markitdown` (via the pptx skill's standard process) to extract text from the produced `.pptx` and verify content accuracy.

### Step 2: 411-Specific Checks

Run each of these checks and report results:

1. **Leadline horizontal story test** — Read all leadlines in order. Do they tell a coherent, connected argument? Flag any that break the logical flow.
2. **Implication quality** — Every callout item must be a strategic conclusion, not a data restatement. Flag items that just repeat what the chart shows without saying "so what."
3. **Source line coverage** — Every data slide must have a source line. Flag any missing.
4. **Sensitivity slide compliance** — Every slide in `sensitivity_slides` must show a range of outcomes. Flag any that show only a single point estimate.
5. **Recommendation slide completeness** — Must have: recommendation statement, supporting reasons, key risks. Flag missing elements.

### Step 3: Fix and Verify

For each flagged issue:
1. Fix the issue in the `.pptx`
2. Re-extract and verify the fix
3. Report what was fixed

Repeat until all checks pass.

### Step 4: Final Report

Present to the user:
- Total slides produced
- QA checks passed/failed
- Any issues that were auto-fixed
- Path to the final `.pptx` file
